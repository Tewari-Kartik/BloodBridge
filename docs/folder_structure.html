End-to-End Data Flow:

📱 User types message on frontend/
↓
🌐 backend/app/api/ receives HTTP request
↓
⚙️ backend/app/services/ orchestrates the pipeline call
↓
🧹 backend/ml/preprocessing/ → cleans text
↓
🚨 backend/ml/triage/ → classifies urgency (P0–P3)
↓
🏷️ backend/ml/ner/ → extracts blood group, hospital, etc.
↓
🎯 backend/ml/matching/ → finds & ranks nearby donors
↓
📲 Response → frontend/ shows urgency + donors on map

Backend — API Server

backend/app/

The FastAPI web server. Receives requests from the frontend, calls the ML pipeline, returns JSON responses.

📂 api/
Route Handlers — One file per group of API endpoints.
routes_request.py, routes_donor.py, routes_forecast.py, routes_health.py
📂 schemas/
Data Shapes — Pydantic models defining exact structure of API input/output.
request.py, donor.py, forecast.py
📂 services/
Business Logic — Orchestrates ML pipeline calls. The "kitchen manager."
request_service.py, donor_service.py, forecast_service.py
📂 db/
Database Layer — SQLAlchemy connection + ORM table definitions.
database.py, models.py
API Endpoint File What It Does
POST /api/request routes_request.py Submit blood request → get urgency + entities + matched donors
GET /api/donors routes_donor.py Search donors by blood group, location, availability
POST /api/donors routes_donor.py Register a new donor
GET /api/forecast routes_forecast.py Get 14-day demand prediction for a city + blood group
🧠

Backend — ML Pipeline (The AI Brain)

backend/ml/

Each subfolder = one stage of the 5-stage pipeline. This is where all the AI magic happens.

🧹 preprocessing/
Stage 1: NLP Text Cleaning — Turn messy WhatsApp messages into clean, model-ready text.
cleaner.py, transliterator.py, language_detect.py, tokenizer.py
"Urgent!! Need O -ve blood 🙏🙏"
→
["urgent", "need", "O-", "blood"]
🚨 triage/
Stage 2: Urgency Classification — Fine-tuned MuRIL transformer. Classifies P0 (critical) → P3 (info).
classifier.py, train.py, evaluate.py
Cleaned token sequence
→
{"urgency": "P0_CRITICAL", "confidence": 0.94}
🏷️ ner/
Stage 3: Named Entity Recognition — Transformer + BiLSTM + CRF extracts structured entities.
extractor.py, train.py, evaluate.py
"Need 3 units O- at AIIMS Delhi"
→
{"blood_group": "O-", "hospital": "AIIMS Delhi", "units": 3}
🎯 matching/
Stage 4: Donor Matching Engine — Filters compatible donors, ranks by proximity + reliability using XGBoost.
compatibility.py, geospatial.py, ranker.py, engine.py
{"blood_group": "O-", "lat": 28.57}
→
[{donor: "Ravi", dist: 2.3km, score: 0.91}]
📈 forecasting/
Stage 5: Demand Forecasting — Temporal Fusion Transformer predicts blood demand 14 days ahead.
tft_model.py, train.py, evaluate.py
Last 90 days demand data
→
Next 14 days predicted demand (with confidence)
pipeline.py — One file that chains all 5 stages: message → clean → classify → extract → match → return. This is the single entry point the API calls.

Frontend — React + TypeScript

frontend/src/

The user-facing application built with Vite + React. Everything the user sees and interacts with.

📂 pages/
Full Page Views — Each file = one route/screen in the app.
Landing.tsx, SOSRequest.tsx, Dashboard.tsx, Forecast.tsx, DonorRegister.tsx
📂 components/
Reusable UI Pieces — Navbar, DonorCard, UrgencyBadge, DonorMap, Charts, Modals.
layout/, request/, donor/, dashboard/, common/
📂 hooks/
Custom React Hooks — Reusable logic for API calls and state management.
useBloodRequest.ts, useDonorSearch.ts, useForecast.ts, useWebSocket.ts
📂 services/
API Communication — Axios calls to FastAPI backend. One file per resource.
api.ts, requestApi.ts, donorApi.ts, forecastApi.ts
📂 types/
TypeScript Interfaces — Mirror of backend Pydantic schemas for type safety.
request.ts, donor.ts, forecast.ts
📂 styles/
CSS — Global theme (dark mode, colors, fonts), animations, component styles.
globals.css, animations.css, components.css
Page What User Sees
Landing.tsx Hero section, problem statement, "Submit SOS" CTA, how-it-works animation
SOSRequest.tsx Text area to paste blood request → real-time pipeline animation → donors on map
Dashboard.tsx Live request feed, stats cards, blood stock chart, demand heatmap
Forecast.tsx 14-day demand charts per blood group, city selector, confidence bands
DonorRegister.tsx Donor sign-up form, auto-detect GPS location, availability toggle

Data, Notebooks & Support

📂 data/raw/
Original unprocessed data — Twitter scrapes, WhatsApp forwards, Kaggle downloads
📂 data/processed/
Output of Stage 1 preprocessing — cleaned messages ready for labeling
📂 data/annotated/
Manually labeled training data — urgency labels (P0–P3), NER BIO tags
📂 data/synthetic/
AI-generated data — GPT/Gemini synthetic messages, 50K simulated donors
📂 notebooks/
Jupyter notebooks for EDA, model development, evaluation — not production code
📂 configs/
YAML files — model hyperparameters, training settings. Change configs, not code.
📂 backend/models/
Saved trained weights — populated after running train.py scripts (initially empty)
📂 docker/
Dockerfiles for containerized deployment (build last)
📂 docs/
Architecture diagrams, API documentation, final results report
